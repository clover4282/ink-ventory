# Ink-ventory 기능 정의서

- 문서 버전: 1.2
- 기준일: 2026-07-15
- 기준 구현: Ruby on Rails 8.1 + SQLite 단일 애플리케이션
- 대상 언어·통화·시간대: 한국어, KRW, Asia/Seoul

## 1. 문서 목적

Ink-ventory의 사용자 기능, 수집 규칙, 알림 규칙과 운영 기준을 현재 코드 기준으로 정의한다. 이 문서는 기능 변경의 기준 문서이며, 사용자에게 보이는 동작이나 업무 규칙을 바꿀 때 코드보다 먼저 수정한다.

상태 표기는 다음과 같다.

- `구현`: 현재 사용자 또는 운영자가 사용할 수 있다.
- `부분 구현`: 내부 코드가 있으나 현재 기본 사용자 화면에서 완결된 흐름으로 제공되지 않는다.
- `제외`: 현재 제품 범위에 포함하지 않는다.

## 2. 서비스 목표와 범위

### 2.1 목표

베스트펜, 블루블랙, 펜갤러리아, 펜로그에서 판매하는 만년필의 공개 가격·재고 정보를 주기적으로 확인하고 다음 기능을 제공한다.

- 비로그인 사용자도 수집된 만년필을 검색·비교한다.
- 로그인하고 알림 이메일을 인증한 사용자는 관심 상품을 저장한다.
- 관심 상품의 재입고·품절·가격 변화를 이메일로 받는다.
- 운영자는 쇼핑몰 수집 상태, 오류와 메일 대기열을 관리한다.

### 2.2 지원 쇼핑몰

| 코드 | 표시 이름 | 대표 호스트 | 쇼핑몰 유형 |
|---|---|---|---|
| `bestpen` | 베스트펜 | `bestpen.kr` | MakeShop |
| `blueblack` | 블루블랙 | `blueblack.co.kr` | Cafe24 |
| `pengalleria` | 펜갤러리아 | `pengalleria.com` | MakeShop |
| `penlog` | 펜로그 | `myungdongmall.com`, `penlog.co.kr` | Cafe24 |

PC·모바일 호스트는 하나의 정규 상품 URL로 통합한다. 같은 정규 URL은 사용자 수와 관계없이 한 번만 수집한다.

### 2.3 사용자 구분

| 구분 | 권한 |
|---|---|
| 비로그인 사용자 | 공개 상품 조회, 실시간 로컬 검색, 필터·정렬, 전체 상품 변동 내역 조회, 판매처 링크 이동 |
| 로그인 사용자 | 비로그인 기능, 알림 이메일 등록·인증 |
| 이메일 인증 사용자 | 좋아요·관심 상품 구독, 이메일 알림 수신 |
| 관리자 | 사이트 수집 중지·재개, 수집 오류·메일 대기열 조회 |

## 3. 공개 메인 카탈로그

### 3.1 상품 표시 — `구현`

정상 수집 이력이 있는 만년필을 로그인 여부와 관계없이 메인 화면에 표시한다.

상품 카드 표시 항목은 다음과 같다.

- 상품 이미지. 이미지가 없으면 `이미지 수집 중` 표시
- 판매처 이름과 판매처 로고
- 상품명과 Ink-ventory 상품 상세 링크
- 기본 판매 가격
- 전체 상품 재고 상태: `재고 있음`, `품절`, `판매 종료`, `확인 중`
- 마지막 정상 확인 시각
- 좋아요 수, 상품 상세 조회수
- Ink-ventory 상품 DB 최초 등록 후 30일 동안 표시되는 `NEW` 태그
- 최근 10일 내 재입고 여부

한 화면에는 검색·필터·정렬 결과 중 최대 50개를 표시한다.

상품명은 카드에서 최대 두 줄로 표시하고, 이후 가격·확인 시각·관심 영역은 같은 행의 카드끼리 같은 높이에 맞춘다. 두 줄을 넘는 상품명은 말줄임 처리하며 상세 페이지에서는 전체 상품명을 표시한다.

상품 카드의 이미지·상품명 또는 카드 여백을 누르면 Ink-ventory 상품 상세 페이지로 이동한다. 좋아요 등 카드 내부 조작 버튼은 상세 이동과 독립적으로 동작한다.

### 3.2 즉시 검색 — `구현`

메인 검색은 서버에 매 글자마다 요청하지 않는다. 최초 페이지 응답에 포함된 수집 상품 데이터를 브라우저에서 검색하므로 입력 즉시 결과가 갱신되고 입력 포커스를 유지한다.

검색 대상은 상품명과 판매처 이름이다.

한국어 입력 규칙은 다음과 같다.

- 문자열을 Unicode NFC로 정규화한다.
- 한글 IME 조합 중에는 입력값을 강제로 변환하지 않고 조합 완료 후 검색한다.
- 공백으로 분리된 모든 검색 단어가 상품명 또는 판매처에서 일치해야 한다.
- 완전한 문구 포함을 가장 높은 우선순위로 둔다.
- 정확한 단어 포함을 그다음 우선순위로 둔다.
- 작은 오타의 유사 일치를 마지막 우선순위로 둔다.
- 정확히 포함된 단어만 강조 표시하고 유사 일치 단어는 강조하지 않는다.
- 검색어는 새 페이지 이동 없이 현재 URL의 `q` 파라미터에 반영한다.

현재 유사 검색은 Unicode NFD 문자 기준 편집 거리를 사용한다.

| 검색어 길이 | 허용 편집 거리 |
|---|---:|
| 1~2자 | 0 |
| 3~5자 | 1 |
| 6자 이상 | 2 |

검색 점수는 완전 문구 포함 `+100`, 정확 단어 포함 `+10`, 유사 단어 일치 `+1`로 계산한다. 검색 정확도순에서는 점수가 높은 상품을 먼저 표시한다.

### 3.3 필터 — `구현`

다음 조건을 함께 적용할 수 있다.

- 판매처
- 재고 있음 또는 품절
- 최소 가격
- 최대 가격
- 최근 10일 내 재입고

가격이 수집되지 않은 상품은 최소·최대 가격 조건이 설정되면 결과에서 제외한다.

### 3.4 정렬 — `구현`

| 정렬 | 기준 |
|---|---|
| 기본·검색 정확도순 | 검색 점수, 기존 표시 순서 |
| 인기도순 | 상품 상세 누적 조회수 내림차순 |
| 최신 등록순 | 상품 DB 등록 시각 내림차순 |
| 좋아요순 | 누적 좋아요 수 내림차순 |
| 가격 낮은순 | 기본 판매 가격 오름차순, 가격 미수집 상품은 마지막 |
| 가격 높은순 | 기본 판매 가격 내림차순, 가격 미수집 상품은 마지막 |
| 최근 재입고순 | 최근 10일 내 마지막 재입고 시각 내림차순 |

검색 점수는 다른 정렬의 동점 순위에도 반영한다.

### 3.5 조회수 집계 — `구현`

- 상품 상세 페이지를 열 때 해당 상품의 조회수를 1 증가시킨다.
- 로그인 여부와 관계없이 집계한다.
- 조회수는 인기도순 정렬에 사용한다.
- 상품 상세를 본 뒤 브라우저 뒤로가기로 목록에 돌아오면 새로고침 없이 해당 카드의 증가한 조회수를 표시한다.

### 3.6 상품 상세 — `구현`

비로그인 사용자를 포함한 모든 사용자가 상품 상세 페이지를 볼 수 있다.

- 상품 이미지, 판매처 이름·로고, 상품명 표시
- 상품 DB 최초 등록 후 30일 동안 `NEW` 태그 표시
- 현재 기본 가격, 전체 재고 상태, 마지막 정상 확인 시각과 누적 조회수 표시
- 목록과 동일한 관심 등록·해제 버튼과 로그인·이메일 인증 상태별 안내 표시
- 관심 등록 시 재입고·품절·가격 변경을 즉시 인증 이메일로 알린다는 설명 표시
- 원본 쇼핑몰의 `판매처에서 구매하기` 링크 제공
- 구매 링크는 HTTPS이면서 지원 쇼핑몰 호스트인 경우에만 표시
- 파서 판정 버전이 기록된 확정 `재입고`, `품절`, `가격 변경` 이력을 최신순으로 표시
- 각 변경 이력에 발생 시각, 이전 값과 현재 값 표시
- 변경 이력이 없으면 아직 확인된 변화가 없다는 안내 표시

### 3.7 공개 변동 내역 — `구현`

로그인 여부와 관계없이 상단의 `변동 내역` 메뉴에서 전체 상품의 최근 확정 변화를 확인한다.

- 파서 판정 버전이 기록된 `재입고`, `품절`, `가격 변경`만 표시
- 최신 발생 순으로 최대 100건 표시
- 상품 이미지, 판매처, 상품명, 변경 종류, 이전 값과 현재 값, 발생 시각 표시
- 상품을 누르면 Ink-ventory 상품 상세 페이지로 이동
- 표시할 확정 변화가 없으면 빈 상태 안내 표시

## 4. 로그인과 알림 이메일

### 4.1 로그인 — `구현`

- 사용자는 이메일 주소로 6자리 인증번호를 받아 로그인한다.
- 인증번호는 발급 후 10분 동안 한 번만 사용할 수 있다.
- 인증번호를 5회 잘못 입력하면 해당 이메일 로그인을 1시간 잠그고, 잠금 해제 후 새 인증번호를 받을 수 있다.
- 같은 이메일의 인증번호 재발급은 60초 후 가능하다.
- 인증 완료 시 로그인과 알림 이메일 인증을 함께 완료한다.
- 로그인 세션은 30일 동안 유지되어 컴퓨터와 브라우저를 재시작해도 로그인 상태를 복원한다.
- 로그인 인증번호 메일은 크롤링 작업과 분리된 메일 전용 큐에서 우선 처리한다.
- 개발 환경에서는 별도의 개발자 모드 로그인을 제공한다.

### 4.2 알림 이메일 인증 — `구현`

- 로그인 이메일을 재입고·품절·가격 알림 수신 주소로 사용한다.
- 이메일 주소를 변경하면 기존 인증을 취소하고 다시 인증한다.
- 인증 링크를 사용하면 주소를 인증하고 알림을 활성화한다.
- 미인증 사용자는 좋아요와 관심 상품 등록을 사용할 수 없다.
- 전체 수신 중지 링크는 해당 주소의 모든 이메일 알림을 비활성화한다.

## 5. 좋아요와 관심 상품

### 5.1 좋아요 — `구현`

- 비로그인 사용자가 하트를 누르면 `좋아요는 로그인 후 이용할 수 있습니다.` 안내를 표시한다.
- 로그인했지만 이메일이 미인증이면 이메일 인증 안내를 표시한다.
- 이메일 인증 사용자가 좋아요를 누르면 좋아요와 상품 구독을 함께 생성한다.
- 생성된 기본 구독은 활성 상태이며 재입고·품절·일반 가격 변화 알림 대상이다.
- 좋아요를 다시 누르면 좋아요와 해당 상품 구독을 함께 삭제한다.
- 좋아요 수는 상품 카드와 좋아요순 정렬에 즉시 반영한다.
- 사용자 한 명은 활성 관심 상품을 최대 10개까지 등록할 수 있다.
- 일시정지한 관심 상품은 제한에서 제외하며, 관심 해제·삭제하면 즉시 새 상품을 등록할 수 있다.
- 이미 활성 관심 상품이 10개를 넘는 사용자는 기존 항목을 유지하되 새 등록과 일시정지 항목의 재활성화를 차단한다.
- 한도에 도달하면 `관심 상품은 최대 10개까지 등록할 수 있습니다.`라고 안내한다.

### 5.2 관심 상품 설정 — `구현`

로그인 사용자는 상단의 `관심 상품` 메뉴에서 별도 관리 페이지로 이동한다. 관리 페이지에서 다음을 제공한다.

- 활성 관심 상품 수와 `현재 수/10` 표시
- 대표 이미지, 판매처, 재고, 가격을 포함한 관심 상품 목록
- 관심 상품 삭제만 제공하며 별도의 알림 활성화 체크박스와 저장 버튼은 두지 않음
- 최근 변경 이력 최대 20건
- 알림 이메일 상태 확인

### 5.3 URL 등록과 쇼핑몰 원격 검색 — `부분 구현`

- 지원 쇼핑몰 URL을 검증·정규화하여 구독하는 서버 동작은 남아 있다.
- 네 쇼핑몰에 원격 검색을 요청하고 검색어를 감시하는 서버 동작과 결과 화면은 남아 있다.
- 현재 메인 화면에는 URL 등록 폼과 원격 통합검색 폼을 노출하지 않는다.
- 메인 검색은 이미 수집된 상품만 즉시 검색한다.

다시 사용자 기능으로 제공하려면 이 문서에서 화면 위치, 등록 조건과 사용자 흐름을 먼저 확정한다.

## 6. 자동 상품 발견과 수집

### 6.1 만년필 전용 발견 — `구현`

- 매일 01:00 KST에 네 쇼핑몰의 만년필 카테고리 또는 만년필 검색 결과를 확인한다.
- 상품명에 `만년필` 또는 `fountain pen`이 명시된 상품을 우선 수집한다.
- 신뢰된 만년필 카테고리에서는 볼펜, 샤프, 롤러볼, 잉크, 컨버터, 카트리지, 닙, 케이스, 리필 등 제외 키워드가 없는 상품도 허용한다.
- 만년필 외 상품과 만년필 액세서리는 수집 대상에서 제외한다.
- 새로 발견한 상품은 다음 수집 대상으로 등록하며, 발견 작업 자체가 사용자에게 자동 구독을 만들지는 않는다.

### 6.2 상품 수집 항목 — `구현`

상품 페이지에서 다음 공개 정보만 수집한다.

- 상품명
- 통화
- 기본 판매 가격
- 상품 전체 재고 상태
- 대표 이미지 URL
- 정규 상품 URL과 판매처 상품 ID
- 확인 시각과 HTTP 캐시 검증값

JSON-LD와 화면에 표시되는 HTML을 사용한다. 상품 설명과 이미지 파일 자체는 저장하지 않고 이미지 원본 URL만 저장한다.

동일 상품은 상품명이 아니라 판매처와 판매처 상품 번호로 식별한다. Cafe24는 `product_no`, MakeShop은 `branduid`를 사용하며 PC·모바일 주소를 하나의 정규 URL로 통합한다. DB는 정규 URL과 `(판매처, 상품 번호)` 조합의 중복을 허용하지 않는다. 같은 상품을 다시 수집하면 새 상품을 만들지 않고 기존 상품에 관측 기록을 추가한다.

### 6.3 옵션·수량 처리 — `제외`

- 닙, 색상 등 옵션 목록을 파싱·저장·전송·표시하지 않는다.
- 옵션별 가격과 옵션별 재고를 수집하지 않는다.
- 화면 공개 수량과 숨겨진 수량을 수집하지 않는다.
- 장바구니 시험으로 재고를 추정하지 않는다.
- `VISIBLE_QUANTITY_CHANGED` 이벤트 이름은 기존 호환을 위해 남아 있지만 현재 생성하지 않는다.

### 6.4 수집 주기와 변경 확정 — `구현`

- 매분 기한이 된 상품을 최대 100개씩 순차 처리한다.
- 상품별 정상 감시 간격 목표는 10분이다.
- 등록량과 사이트 백오프에 따라 실제 간격은 10분보다 길어질 수 있다.
- 최초 정상 수집은 기준 상태만 저장하고 변경 이벤트나 알림을 만들지 않는다.
- 모든 정상 수집은 재고·가격 상태와 파서 판정 버전을 상품 관측 기록에 저장한다.
- 파서 판정 버전이 바뀐 첫 정상 수집은 새로운 기준 상태로 저장하고 재입고·품절·가격 변경 이벤트를 만들지 않는다.
- 기준 상태와 다른 값을 처음 보면 임시 상태로 저장하고 약 2분 뒤 재확인한다.
- 최초 변경 관측 후 1분 이상 지나 같은 값이 다시 확인될 때 변경을 확정한다.
- 값이 기존 상태로 돌아오거나 임시 상태와 달라지면 이벤트를 확정하지 않는다.
- 같은 파서 판정 버전 안에서 확정된 재고·가격 변화만 변경 이벤트로 장기 보관한다.
- 변경 이벤트에는 해당 관측의 파서 판정 버전을 함께 기록한다.
- 파서 판정 버전 도입 전 이벤트는 삭제하지 않지만 확정 이력과 최근 재입고 배지에서는 제외한다.

### 6.5 요청 제한과 장애 처리 — `구현`

- 사이트별 요청은 동시에 하나만 수행한다.
- 같은 사이트 요청 간 최소 2초와 최대 0.4초 임의 지터를 적용한다.
- 타임아웃, 네트워크 오류와 파싱 실패는 품절로 처리하지 않고 마지막 정상 상태를 유지한다.
- `403`, `429`, 반복 `5xx`는 사이트별 지수 백오프를 적용하며 최대 24시간까지 중지한다.
- 404가 3회 연속 확인되면 상품을 `판매 종료`로 변경하고 `REMOVED` 이벤트를 만든다.
- 관리자는 사이트별 긴급 수집 중지와 재개를 할 수 있다.

## 7. 변경 이벤트와 알림

### 7.1 이벤트

| 이벤트 | 생성 조건 | 발송 방식 | 상태 |
|---|---|---|---|
| `RESTOCKED` | 이전 상태가 재고 있음이 아니고 현재 재고 있음 | 즉시 | 구현 |
| `SOLD_OUT` | 재고 있음에서 품절로 변경 | 즉시 | 구현 |
| `PRICE_CHANGED` | 이전·현재 가격이 모두 있고 값이 변경 | 즉시 | 구현 |
| `NEW_SEARCH_RESULT` | 감시 검색어에서 기존에 없던 후보 발견 | 메일 없음 | 부분 구현 |
| `REMOVED` | 404가 3회 연속 확인 | 메일 없음 | 구현 |
| `VISIBLE_QUANTITY_CHANGED` | 공개 수량 변경 | 해당 없음 | 제외 |

목표 가격 설정과 목표가 도달 판정은 v1에서 사용하지 않는다. 기존 DB 컬럼과 과거 `TARGET_REACHED` 기록은 호환성을 위해 보존하지만 신규 이벤트를 생성하지 않는다.

### 7.2 이메일 종류 — `구현`

- 이메일 주소 인증 메일
- 재입고·품절·가격 변경 즉시 메일

변경 알림 메일은 상품명, 판매처, 이전·현재 값, 발생 시각, 원본 구매 링크, 해당 상품 구독 해제 링크와 전체 수신 중지 링크를 제공한다.

### 7.3 발송 안정성 — `구현`

- 발송 요청을 SQLite 기반 메일 대기열에 먼저 저장한다.
- 사용자·이벤트 또는 사용자·날짜 단위 중복 키로 같은 메일의 중복 생성을 막는다.
- 발송 직전에 인증·수신 동의·현재 이메일 주소를 다시 확인하고, 조건이 바뀐 대기 메일은 취소한다.
- 실패 메일은 지수 백오프로 다시 시도하며 최대 재시도 간격은 24시간이다.
- 30분 이상 처리 중인 메일은 실패 상태로 되돌려 재시도한다.
- 개발 환경은 SMTP 자격 증명이 없으면 메일을 `tmp/mails`에 저장하고, `.env`에 SES SMTP 자격 증명이 있으면 실제 발송한다.
- 운영 환경은 Amazon SES SMTP 서울 리전을 사용한다.

## 8. 운영자와 계정 관리

### 8.1 관리자 화면 — `구현`

- 사이트별 수집 활성화 상태, 연속 실패 횟수와 백오프 시각 조회
- 사이트별 긴급 중지·수집 재개
- 연속 오류가 있는 상품 최대 50개 조회
- 최근 메일 대기열 최대 50개와 상태·시도 횟수·오류 조회

### 8.2 회원 탈퇴 — `구현`

회원 탈퇴 시 사용자와 연결된 다음 정보를 삭제한다.

- 로그인 이메일
- 알림 이메일
- 좋아요와 구독
- 검색어 감시와 검색 시도
- 개인 발송 기록과 이벤트 수신 기록

전역 상품과 개인을 식별하지 않는 가격·재고 관측 이력은 유지한다.

### 8.3 보관 기간 — `구현`

- 완료·실패 메일 발송 로그: 마지막 변경 후 30일
- 상품 관측 기록: 1년
- 확정된 재고·가격 변경 이벤트: 별도 삭제 정책을 정하기 전까지 계속 보관
- 검색 요청 제한 기록: 1일 이내

## 9. 보안과 수집 준수

### 9.1 URL·네트워크 보안 — `구현`

- HTTPS와 지정된 네 쇼핑몰 호스트만 허용한다.
- 상품 ID를 확인한 후 표준 상품 URL로 정규화한다.
- 최초 요청과 모든 리디렉션에서 허용 호스트를 다시 검사한다.
- DNS의 모든 응답 IP가 공인 주소일 때만 접속하여 SSRF를 차단한다.
- 응답 본문은 최대 5MB로 제한한다.
- 접속·읽기 타임아웃을 적용한다.
- 사용자에게 제공하는 원본 구매 링크도 HTTPS와 지정된 쇼핑몰 호스트를 다시 검사하며, 검사를 통과하지 못하면 링크를 표시하지 않는다.

### 9.2 운영 전 준수 사항

- 각 쇼핑몰의 robots 정책, 이용약관과 자동 조회 허용 범위를 재확인한다.
- 실제 서비스 이름과 연락처가 포함된 `CRAWLER_USER_AGENT`를 설정한다.
- 개인정보 처리방침, 이용약관과 이메일 알림 수신 조건을 실제 운영자 정보로 확정한다.
- SES 발신 도메인 SPF·DKIM·DMARC와 프로덕션 발송 승인을 완료한다.

## 10. 시스템 구성과 일정

### 10.1 구성 — `구현`

- Ruby 3.4, Rails 8.1
- 서버 렌더링 HTML과 기본 JavaScript
- SQLite 애플리케이션 DB와 SQLite 기반 Solid Queue
- 한 컨테이너 안에서 메일·기본 작업 워커와 크롤링 워커를 분리해 인증 메일이 수집 작업 때문에 지연되지 않게 한다.
- `Net::HTTP` + Nokogiri 수집기
- 개발 환경: Rails와 작업 처리기가 한 컨테이너에서 동작
- 운영 환경: Caddy HTTPS 컨테이너 + Rails 애플리케이션 컨테이너
- 공개 JSON API 없음

### 10.2 정기 작업

| KST 일정 | 작업 |
|---|---|
| 매분 | 기한이 된 상품 수집·변경 재확인 |
| 매분 | 즉시 메일·실패 메일 발송 처리 |
| 매일 01:00 | 네 쇼핑몰 만년필 신규 상품 발견 |
| 매일 02:00 | 저장된 검색어 신규 후보 확인 |
| 매일 03:00 | 오래된 발송·관측·검색 제한 기록 정리 |
| 매시 12분 | 완료된 Solid Queue 작업 정리 |

### 10.3 배포 준비 기준

- CI와 운영 DB 준비 명령은 PostgreSQL 서비스 없이 SQLite 구성으로 실행한다.
- Docker 빌드 컨텍스트에는 `.env`, SQLite 데이터, 브라우저 테스트 산출물과 로컬 편집기 설정을 포함하지 않는다.
- 개발과 운영 Docker Compose는 서로 다른 프로젝트 이름과 SQLite 볼륨을 사용해 동시에 실행해도 컨테이너나 데이터가 교체되지 않게 한다.
- 운영 전 64바이트 이상의 임의 `SECRET_KEY_BASE`, 실제 공개 호스트, SES SMTP 자격 증명과 인증된 발신 주소를 설정한다.
- 개인정보 처리방침에 실제 운영자 또는 서비스 책임 주체, 수신 가능한 공개 연락처와 호스팅 위탁 사업자를 기재한다.
- `CRAWLER_USER_AGENT`에는 서비스 이름과 수신 가능한 공개 연락처를 포함한다.
- 최초 배포 후 사이트 기본 데이터와 만년필 카탈로그 발견 작업을 실행하고 `/up`, 로그인 메일, 수집 작업과 백업 생성을 확인한다.

## 12. 출시 확인 기준

- 네 쇼핑몰의 만년필만 자동 발견되고 액세서리와 다른 필기구는 제외된다.
- 비로그인 사용자가 메인 화면에서 상품 이미지, 판매처 로고, 가격과 재고를 볼 수 있다.
- 상품 DB 최초 등록 후 30일 동안 상품 카드에 `NEW` 태그가 표시되고, 30일이 지나면 표시되지 않는다.
- 상품 카드를 누르면 공개 상세 페이지로 이동하고 조회수가 증가한다.
- 상세 페이지에서 뒤로 돌아온 목록 카드에도 증가한 조회수가 즉시 반영된다.
- 상품 상세에서 현재 상태와 확정된 재입고·품절·가격 변경 이력을 확인할 수 있다.
- 상품 상세에서 30일 이내 신규 등록 태그와 관심 버튼·알림 방식을 확인할 수 있다.
- 비로그인 사용자가 공개 변동 내역에서 최근 확정된 재입고·품절·가격 변경을 최신순으로 확인하고 상품 상세로 이동할 수 있다.
- 한글 조합 중 입력 포커스를 잃지 않고 검색 결과가 즉시 갱신된다.
- 띄어쓰기와 작은 오타가 있어도 유사 상품이 검색되며 정확한 일치가 먼저 나온다.
- 정확히 일치한 단어만 강조 표시된다.
- 판매처·재고·가격·최근 재입고 필터와 모든 정렬 기준이 올바르게 적용된다.
- 비로그인 좋아요는 로그인 안내를 표시하고 데이터는 변경하지 않는다.
- 인증 사용자의 좋아요가 구독과 함께 생성·삭제된다.
- 활성 관심 상품은 사용자당 최대 10개로 제한되며 한도 초과 시 기존 관심 상품과 좋아요 수가 변경되지 않는다.
- 로그인 사용자가 별도 관심 상품 관리 페이지에서 `현재 수/10`, 상품 이미지·상태·가격, 삭제와 최근 변경 이력을 확인할 수 있다.
- 관심 상품 카드에는 알림 활성화 체크박스나 저장 버튼이 표시되지 않는다.
- 같은 상품을 여러 사용자가 좋아요해도 상품 수집은 한 번만 수행한다.
- 최초 수집 또는 한 번만 관측된 변경에는 알림이 생성되지 않는다.
- 파서 판정 버전 변경은 새 기준값만 저장하며 재고·가격 변경으로 기록되지 않는다.
- 같은 파서 판정 버전에서 재확인된 재고·가격 변화는 이전 값, 현재 값과 발생 시각을 보존한다.
- 파서 판정 버전이 없는 기존 이벤트는 보존하되 상세 이력과 최근 재입고 배지에 표시하지 않는다.
- 재입고·품절·가격 변경은 올바른 사용자에게만 즉시 발송된다.
- 가격이 변해도 목표가 도달 이벤트는 생성하지 않는다.
- 판매 종료·신규 검색 결과는 이벤트만 저장하고 메일을 생성하지 않는다.
- 수집 실패는 품절로 바뀌지 않고, 같은 이벤트·사용자 메일은 중복 발송되지 않는다.
- 전체 Rails 테스트, 정적 검사와 Docker 개발 환경 스모크 테스트가 통과한다.
- 상세 페이지의 구매 링크가 허용된 쇼핑몰 HTTPS 주소에서만 표시된다.
- GitHub Actions가 별도 PostgreSQL 없이 SQLite DB를 준비해 전체 테스트를 통과한다.
- Docker 운영 이미지 빌드, 운영 환경 부팅, 상태 확인과 SQLite 백업 생성 스모크 테스트가 통과한다.
- 운영 비밀키, 공개 연락처, 크롤러 User-Agent, 개인정보 처리방침의 운영자·위탁 정보가 실제 값으로 확정된다.

## 13. 기능 변경 절차

기능을 추가·변경·삭제할 때 다음 순서를 지킨다.

1. 이 문서에서 대상 기능의 요구사항, 상태와 출시 확인 기준을 먼저 수정한다.
2. 변경된 문서와 기존 코드의 차이를 구현 범위로 확정한다.
3. 재현 또는 인수 테스트를 추가·수정한다.
4. 코드와 DB를 최소 범위로 수정한다.
5. 테스트 결과와 문서·코드 일치 여부를 확인한다.

사용자 동작이나 업무 규칙이 바뀌지 않는 순수 리팩터링·오탈자 수정은 기능 정의서 변경 대상에서 제외한다.
